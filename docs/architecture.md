## ADR-001. Архитектура «Telegram Mini App» вместо бота с командами

**Дата:** 2026-06-29
**Статус:** принято

### Контекст
- Проект — семейный планировщик на двоих (муж/жена): общие задачи, календарь, события.
- Хостинг — Proxmox LXC 109 на РФ/Selectel-хосте. **`api.telegram.org` с хоста заблокирован**
  (см. `docs/infra.md` и `private/proxmox-notes.md`): любой исходящий запрос к Telegram с сервера
  висит/таймаутит. Значит классические схемы (long polling, синхронная отправка) не работают «в лоб».
- Исходное ТЗ предполагало полноценного бота с командами (`/add`, `/list`, …) и вход на сайт
  по одноразовой ссылке (`/link`).

### Решение
Переходим на модель **«бот = лаунчер, вся логика — в Mini App (веб-приложении)»**:

1. **Бот — минимальный.** Никакой обработки команд в MVP. В @BotFather настраивается Menu Button
   (или кнопка), открывающая Mini App по URL нашего сайта. Это конфигурируется без кода и без
   обращений к Telegram с хоста.
2. **Frontend — Telegram Mini App (SPA).** Открывается внутри Telegram, отдаётся с нашего домена
   (Cloudflare → NPM → LXC 109). Это основной интерфейс. Фокус разработки — здесь.
3. **Авторизация — через `initData`** (заменяет `/link` из ТЗ, раздел 3.3). Mini App передаёт
   фронту подписанные данные `initData` (включая Telegram user id + hash). Бэкенд валидирует
   подпись HMAC-SHA256 с bot-токеном по документированному алгоритму Telegram Mini Apps —
   **локально, без запросов к Telegram**, достаёт user id, сверяет с allowlist (двое) и выдаёт
   свою сессию (cookie/JWT). Одноразовые ссылки не нужны.
4. **Backend — Go + PostgreSQL** (по ТЗ): REST API для Mini App, валидация initData, сессии,
   CRUD задач/событий, общее семейное пространство.
5. **Исходящие уведомления (бот → пользователю)** — через **облачный n8n**: бэкенд шлёт HTTP POST
   на webhook n8n (chat_id, текст), а n8n (у него доступ к Telegram есть) отправляет сообщение.
   Вызов делаем асинхронно и с таймаутом. Локальный n8n на том же хосте использовать нельзя (тоже под блоком).
6. **Приём апдейтов из Telegram — в MVP не нужен.** Всё взаимодействие пользователя идёт внутри
   Mini App обычными HTTPS-запросами фронт↔бэк. Если бот-логика понадобится позже — использовать
   **webhook** (входящий трафик, блок не мешает) или Telegram Trigger в облачном n8n, но **не**
   long polling (`getUpdates` — исходящий, заблокирован).

### Последствия
- ➕ Резко упрощается бот (нет парсинга команд, состояний диалога) → меньше кода и багов.
- ➕ Авторизация проще и безопаснее `/link` (нет временных токенов, подпись проверяется локально).
- ➕ Обходит блок Telegram: для базового потока сервер вообще не ходит в Telegram.
- ➕ Концентрация усилий на фронте (Mini App) — это и есть основной UX.
- ➖ Уведомления зависят от внешнего (облачного) n8n — это отдельная точка отказа, документируем.
- ➖ Требуется валидный публичный HTTPS-домен для Mini App (уже есть: NPM + Let's Encrypt).
- 🔄 ТЗ корректируется: раздел 3.3 (`/link`) → авторизация по initData; раздел 4 (команды бота) →
  вне scope MVP (бот = кнопка запуска).

### Открытые вопросы (решить на этапе реализации)
- Точный алгоритм валидации initData (ключ = HMAC-SHA256("WebAppData", bot_token), затем сверка hash).
- Формат сессии (cookie HttpOnly vs JWT), срок жизни, что делать вне Telegram-клиента (десктоп/браузер).
- Какие именно уведомления нужны (напоминания о событиях) и их триггеры.
- Узнать и зафиксировать telegram user id обоих пользователей для allowlist.
