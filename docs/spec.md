# Техническое задание: семейный Telegram to-do list с календарем и веб-интерфейсом

## 1. Цель проекта

Разработать приложение для совместного ведения задач, дел, встреч и расписания для двух пользователей: мужа и жены.

Основная идея: оба пользователя могут через Telegram-бота и через веб-сайт добавлять, смотреть, переносить, редактировать, завершать и удалять задачи/события. Все данные общие: единый список дел, единый календарь, единое пространство доступа.

## 2. Основные компоненты

Приложение состоит из трех частей:

1. Telegram-бот.
2. Веб-интерфейс.
3. Backend API + база данных.

## 3. Пользователи и доступ

### 3.1. Пользователи

В системе есть семейное пространство, в котором состоят два пользователя:

- пользователь 1;
- пользователь 2.

Оба пользователя имеют доступ к одним и тем же данным:

- общий список задач;
- общий календарь;
- общие встречи;
- общие напоминания;
- общая история изменений, если она будет реализована.

### 3.2. Права доступа

На MVP оба пользователя имеют одинаковые права:

- создавать задачи;
- создавать встречи;
- просматривать задачи и события;
- редактировать задачи и события;
- переносить задачи и события;
- отмечать задачи выполненными;
- удалять задачи и события;
- получать ссылку на веб-интерфейс.

В будущем можно добавить роли, например `owner`, `member`, `viewer`, но в MVP это не обязательно.

### 3.3. Авторизация

#### Telegram

Доступ к боту должен быть разрешен только заранее добавленным Telegram-пользователям.

Для каждого пользователя нужно хранить:

- Telegram user ID;
- имя пользователя;
- username, если есть;
- дату подключения;
- привязку к семейному пространству.

Если боту пишет неизвестный пользователь, бот должен отказать в доступе и показать сообщение вроде:

> У вас нет доступа к этому списку. Попросите владельца добавить вас.

#### Веб-сайт

Веб-интерфейс должен быть доступен только авторизованным пользователям.

Возможные варианты авторизации:

1. Вход по одноразовой ссылке из Telegram.
2. Вход по Telegram Login Widget.
3. Простая авторизация по токену на MVP.

Рекомендуемый вариант для MVP: команда `/link`, которая генерирует временную ссылку на сайт для конкретного пользователя.

Пример:

```text
/link
```

Бот отвечает:

```text
Ваша ссылка для входа: https://app.example.com/auth/telegram-token/abc123
Ссылка действует 15 минут.
```

После входа пользователь попадает в общее семейное пространство.

## 4. Telegram-бот

### 4.1. Общая логика

Бот должен работать:

- в личных сообщениях;
- в общем Telegram-чате, где оба пользователя могут вызывать бота.

Все созданные через бота задачи и события попадают в общий список и общий календарь.

У каждой задачи/встречи нужно сохранять автора создания.

Пример:

```text
Создал: Максим
```

или

```text
Создала: Анна
```

### 4.2. Команды Telegram-бота

#### `/start`

Показывает приветствие и краткую справку.

Пример ответа:

```text
Привет! Я семейный планировщик.

Доступные команды:
/plan — добавить задачу
/meeting — добавить встречу
/list — список задач
/today — дела на сегодня
/week — дела на неделю
/done — отметить задачу выполненной
/delete — удалить задачу
/move — перенести задачу или встречу
/calendar — ближайшие события
/link — открыть сайт
/help — помощь
```

#### `/plan`

Добавляет обычную задачу.

Формат:

```text
/plan Купить продукты завтра 18:00
```

или:

```text
/plan Позвонить врачу
```

Если дата и время не указаны, задача создается без срока выполнения.

Поля задачи:

- название;
- описание, если есть;
- дата и время, если указаны;
- статус;
- автор;
- дата создания;
- дата обновления.

Пример ответа:

```text
Задача добавлена:
Купить продукты
Дата: завтра, 18:00
Создал: Максим
```

#### `/meeting`

Добавляет встречу в календарь.

Формат:

```text
/meeting Врач завтра 12:30
```

или:

```text
/meeting Ужин с друзьями в субботу 19:00
```

Поля встречи:

- название;
- дата начала;
- время начала;
- длительность, если указана;
- место, если указано;
- описание;
- автор;
- дата создания;
- дата обновления.

Если бот не смог точно распознать дату или время, он должен уточнить детали.

Пример:

```text
Я не понял дату встречи. Напишите, пожалуйста, так:
/meeting Врач 30.06.2026 12:30
```

#### `/visit`

Альтернативная команда для добавления визита или похода куда-либо.

Пример:

```text
/visit Стоматолог 02.07.2026 15:00
```

По сути работает как `/meeting`, но тип события сохраняется как `visit`.

#### `/list`

Показывает список активных задач.

Пример ответа:

```text
Активные задачи:

1. Купить продукты — сегодня 18:00
2. Позвонить врачу — без даты
3. Забрать заказ — завтра 12:00
```

Желательно выводить ID или короткий номер задачи, чтобы с ней можно было дальше работать.

#### `/today`

Показывает задачи и события на сегодня.

Пример:

```text
Сегодня:

Задачи:
1. Купить продукты — 18:00

Встречи:
1. Врач — 12:30
```

#### `/week`

Показывает задачи и события на ближайшие 7 дней.

#### `/calendar`

Показывает ближайшие календарные события.

Пример:

```text
Ближайшие события:

1. Врач — завтра 12:30
2. Ужин с друзьями — суббота 19:00
```

#### `/done`

Отмечает задачу выполненной.

Формат:

```text
/done 3
```

Пример ответа:

```text
Задача выполнена: Забрать заказ
```

Нужно сохранить:

- кто отметил задачу выполненной;
- дату и время выполнения.

#### `/delete`

Удаляет задачу или событие.

Формат:

```text
/delete task 3
/delete event 2
```

На MVP можно сделать мягкое удаление: объект получает статус `deleted`, но физически из базы не удаляется.

#### `/move`

Переносит задачу или событие.

Формат:

```text
/move task 3 завтра 15:00
/move event 2 01.07.2026 19:00
```

Пример ответа:

```text
Перенесено:
Забрать заказ
Новая дата: завтра, 15:00
```

#### `/edit`

Редактирует название или описание задачи/события.

Формат:

```text
/edit task 3 Купить продукты и корм для собаки
```

#### `/link`

Генерирует ссылку на веб-интерфейс.

Пример ответа:

```text
Ссылка для входа на сайт:
https://app.example.com/auth/telegram-token/abc123

Ссылка действует 15 минут.
```

#### `/help`

Показывает список команд и примеры использования.

## 5. Веб-интерфейс

### 5.1. Общая логика

Веб-сайт должен позволять обоим пользователям работать с общим семейным пространством.

После входа пользователь видит:

- общий календарь;
- общий список дел;
- ближайшие события;
- задачи без даты;
- выполненные задачи, если включен фильтр.

### 5.2. Основные вкладки

#### Вкладка `Календарь`

Функции:

- просмотр задач и встреч по дням, неделям и месяцам;
- создание события;
- создание задачи с датой;
- перенос события drag-and-drop;
- редактирование события;
- удаление события;
- отметка задачи выполненной прямо из календаря;
- фильтрация по типу: задачи, встречи, визиты.

#### Вкладка `Список дел`

Функции:

- просмотр всех активных задач;
- создание задачи;
- редактирование задачи;
- отметка выполненной;
- удаление;
- назначение даты и времени;
- фильтр по статусу;
- фильтр по автору;
- сортировка по дате создания или дедлайну.

#### Вкладка `Сегодня`

Показывает краткий список дел на текущий день:

- задачи на сегодня;
- встречи на сегодня;
- просроченные задачи;
- задачи без даты.

#### Вкладка `Настройки`

На MVP минимум:

- список пользователей семейного пространства;
- Telegram ID подключенных пользователей;
- настройка часового пояса;
- настройка уведомлений.

## 6. Основные сущности

### 6.1. Family Workspace

Общее пространство пользователей.

Поля:

- id;
- name;
- created_at;
- updated_at.

### 6.2. User

Поля:

- id;
- workspace_id;
- telegram_user_id;
- telegram_username;
- display_name;
- role;
- created_at;
- updated_at.

### 6.3. Task

Поля:

- id;
- workspace_id;
- title;
- description;
- status: `active`, `done`, `deleted`;
- due_at;
- created_by_user_id;
- completed_by_user_id;
- completed_at;
- deleted_at;
- created_at;
- updated_at.

### 6.4. Event

Поля:

- id;
- workspace_id;
- title;
- description;
- type: `meeting`, `visit`, `custom`;
- starts_at;
- ends_at;
- location;
- created_by_user_id;
- status: `active`, `cancelled`, `deleted`;
- created_at;
- updated_at.

### 6.5. AuthToken

Для входа на сайт по ссылке из Telegram.

Поля:

- id;
- user_id;
- token;
- expires_at;
- used_at;
- created_at.

## 7. Уведомления

На MVP желательно реализовать уведомления в Telegram.

Типы уведомлений:

- напоминание о задаче;
- напоминание о встрече;
- уведомление о новой задаче, созданной вторым пользователем;
- уведомление о переносе встречи;
- уведомление о выполнении задачи.

Пример:

```text
Анна добавила задачу:
Купить лекарства
Срок: сегодня 19:00
```

На MVP можно сделать настройку:

- уведомлять обо всех изменениях;
- уведомлять только о напоминаниях;
- не уведомлять.

## 8. Backend API

Примерный список API-методов:

### Auth

- `POST /auth/telegram-token` — вход по токену;
- `POST /auth/logout` — выход.

### Tasks

- `GET /tasks` — список задач;
- `POST /tasks` — создать задачу;
- `GET /tasks/{id}` — получить задачу;
- `PATCH /tasks/{id}` — обновить задачу;
- `POST /tasks/{id}/done` — отметить выполненной;
- `DELETE /tasks/{id}` — удалить задачу.

### Events

- `GET /events` — список событий;
- `POST /events` — создать событие;
- `GET /events/{id}` — получить событие;
- `PATCH /events/{id}` — обновить событие;
- `DELETE /events/{id}` — удалить событие.

### Calendar

- `GET /calendar?from=...&to=...` — получить задачи и события за период.

### Users

- `GET /users` — список пользователей семейного пространства.

## 9. Нефункциональные требования

- Интерфейс должен быть адаптивным: desktop + mobile.
- Все даты и время должны храниться в UTC, отображаться в выбранном часовом поясе.
- Доступ к данным должен быть ограничен workspace_id.
- Пользователь из одного пространства не должен видеть данные другого пространства.
- Все действия должны проверять права доступа.
- Нужно логировать основные действия: создание, изменение, удаление, выполнение.
- Бот должен корректно обрабатывать неизвестные команды и ошибки парсинга даты.

## 10. MVP

В MVP нужно реализовать:

1. Telegram-бота с командами:
   - `/start`;
   - `/plan`;
   - `/meeting`;
   - `/list`;
   - `/today`;
   - `/done`;
   - `/delete`;
   - `/move`;
   - `/link`;
   - `/help`.

2. Общий доступ для двух пользователей.

3. Общую базу задач и событий.

4. Веб-сайт с вкладками:
   - календарь;
   - список дел;
   - сегодня.

5. Возможность через сайт:
   - создать задачу;
   - создать встречу;
   - изменить задачу;
   - изменить встречу;
   - перенести задачу/встречу;
   - отметить задачу выполненной;
   - удалить задачу/встречу.

6. Авторизацию через ссылку из Telegram.

7. Уведомления в Telegram о ближайших событиях и задачах.

## 12. Пример пользовательских сценариев

### Сценарий 1: добавить задачу через Telegram

Пользователь пишет:

```text
/plan Купить продукты завтра 18:00
```

Бот создает задачу в общем списке и отвечает:

```text
Задача добавлена:
Купить продукты
Дата: завтра, 18:00
```

Второй пользователь видит эту задачу на сайте и может отметить ее выполненной.

### Сценарий 2: добавить встречу через Telegram

Пользователь пишет:

```text
/meeting Врач 30.06.2026 12:30
```

Бот добавляет встречу в общий календарь.

Оба пользователя видят встречу на сайте.

### Сценарий 3: открыть сайт

Пользователь пишет:

```text
/link
```

Бот выдает временную ссылку.

Пользователь переходит по ссылке и попадает в общий семейный календарь.

### Сценарий 4: выполнить задачу на сайте

Один пользователь открывает вкладку `Список дел`, нажимает `Выполнено` у задачи.

Задача меняет статус на `done`.

В Telegram может прийти уведомление:

```text
Максим выполнил задачу:
Купить продукты
```

### Сценарий 5: общий чат в Telegram

В общем чате пользователь пишет:

```text
@family_planner_bot /plan Купить корм собаке сегодня 20:00
```

Бот проверяет, что пользователь имеет доступ к семейному пространству, создает задачу и отвечает в чат.


## 13. Техническая архитектура

### 13.1. Рекомендуемый стек

Backend:

- язык: Go;
- HTTP API: стандартный `net/http`, Chi, Gin или Echo;
- Telegram Bot API: библиотека для Go, например `go-telegram-bot-api`;
- база данных: PostgreSQL;
- миграции: Goose, Atlas или golang-migrate;
- конфигурация: env-переменные;
- контейнеризация: Docker + Docker Compose;
- логирование: structured logs, например slog/zap/zerolog;
- тесты: Go test + интеграционные тесты с PostgreSQL через Docker/Testcontainers.

Frontend:

- React/Vue/Svelte или другой SPA-фреймворк;
- календарь: FullCalendar или аналогичная библиотека;
- сборка в Docker image или отдача статики через backend/nginx.

Infrastructure:

- LXC-контейнер в Proxmox;
- внутри LXC: Docker + Docker Compose;
- внешний reverse proxy: Nginx Proxy Manager, который уже используется для связи домена, IP, SSL и проксирования;
- TLS-сертификаты через Nginx Proxy Manager / Let's Encrypt;
- PostgreSQL в Docker Compose внутри LXC для MVP, либо отдельный PostgreSQL/LXC/managed-инстанс в будущем.

### 13.2. Сервисы приложения

Для MVP рекомендуется не усложнять микросервисами и сделать модульный монолит.

Состав сервисов в Docker Compose:

1. `app` — Go backend, внутри которого находятся:
   - HTTP API для сайта;
   - Telegram bot handler;
   - бизнес-логика задач, событий, пользователей и уведомлений;
   - scheduler для напоминаний.

2. `frontend` — веб-интерфейс. Возможны два варианта:
   - отдельный контейнер со статикой и nginx;
   - сборка фронта в статику и отдача через backend/reverse proxy.

3. `postgres` — база данных PostgreSQL.

4. Опционально `migrate` — одноразовый контейнер для применения миграций перед запуском приложения.

Reverse proxy внутри docker-compose не обязателен, если на Proxmox уже есть отдельный Nginx Proxy Manager. В таком случае Nginx Proxy Manager проксирует внешний домен на IP LXC-контейнера и порт приложения, например `http://<lxc-ip>:8080`.

На старте не рекомендуется разделять backend, bot и scheduler на отдельные сервисы, потому что логика и база общие, нагрузка маленькая, а сопровождать один сервис проще. При росте проекта можно вынести Telegram-бот и scheduler отдельно.

### 13.3. PostgreSQL

Да, для хранения данных рекомендуется PostgreSQL.

Причины:

- надежное хранение задач, событий, пользователей и токенов;
- удобная работа с датами, индексами и транзакциями;
- можно делать выборки по календарным диапазонам;
- легко добавлять историю изменений, recurring tasks, категории и теги;
- хорошо подходит для Go-приложений.

Основные таблицы:

- `workspaces`;
- `users`;
- `tasks`;
- `events`;
- `auth_tokens`;
- `notification_settings`;
- `notifications` или `reminders`;
- опционально `audit_log`.

Важные индексы:

- `tasks(workspace_id, status, due_at)`;
- `events(workspace_id, starts_at, ends_at)`;
- `users(telegram_user_id)`;
- `auth_tokens(token, expires_at)`;
- `reminders(send_at, status)`.

Все записи должны быть привязаны к `workspace_id`, чтобы в будущем можно было поддержать несколько семейных пространств.

### 13.4. Модульная структура backend

Пример структуры Go-проекта:

```text
/cmd

    main.go
/internal
  /config
  /http
    /handlers
    /middleware
  /telegram
    /commands
    /parser
  /domain

    /event
    /user
    /workspace
    /reminder
  /repository
    /postgres
  /service
  /scheduler
  /auth
  /logger
/migrations
/web
/docker
```

Основной принцип: Telegram-бот и сайт используют одну и ту же бизнес-логику.

Например:

- Telegram-команда `/plan` вызывает `TaskService.CreateTask`;
- веб-форма создания задачи тоже вызывает `TaskService.CreateTask`;
- перенос задачи через сайт и команда `/move` используют один метод `TaskService.UpdateTaskDate`.

Так получится избежать дублирования логики.


### 13.5. Размещение в Proxmox LXC и Nginx Proxy Manager

Проект будет размещаться не на классической отдельной ВМ, а в LXC-контейнере внутри Proxmox.

Рекомендуемая схема:

```text
Интернет
  ↓
Домен: planner.example.com
  ↓
Nginx Proxy Manager
  ↓
LXC-контейнер Proxmox
  ↓
Docker Compose
  ↓
app + postgres
```

#### LXC-контейнер

В LXC нужно установить:

- Docker;
- Docker Compose plugin;
- Git, если деплой будет через `git pull`;
- базовые утилиты: curl, ca-certificates, openssh-server.

Рекомендуемые ресурсы для MVP:

- CPU: 1-2 vCPU;
- RAM: 1-2 GB;
- disk: от 10-20 GB;
- backup LXC через Proxmox Backup/обычные snapshot'ы;
- отдельный Docker volume для PostgreSQL.

Важно: если Docker запускается внутри LXC, контейнер должен быть настроен так, чтобы Docker корректно работал. В Proxmox для этого обычно нужен privileged LXC или корректно настроенный unprivileged LXC с поддержкой nesting. Для личного проекта проще начать с LXC с включенным nesting.

#### Nginx Proxy Manager

Nginx Proxy Manager используется как внешний reverse proxy и отвечает за:

- привязку домена к приложению;
- выпуск и обновление SSL-сертификата;
- проксирование HTTPS-трафика на LXC;
- настройку WebSocket, если он понадобится для live-обновлений календаря.

Пример настройки Proxy Host в Nginx Proxy Manager:

```text
Domain Names: planner.example.com
Scheme: http
Forward Hostname / IP: <IP LXC-контейнера>
Forward Port: 8080
Websockets Support: enabled, если будут live-обновления
Block Common Exploits: enabled
SSL: Let's Encrypt certificate
Force SSL: enabled
HTTP/2 Support: enabled
```

В приложении переменная `APP_BASE_URL` должна указывать на публичный домен:

```text
APP_BASE_URL=https://planner.example.com
```

Именно этот адрес должен использоваться при генерации ссылок командой `/link`.

## 14. CI/CD pipeline

### 14.1. Цель CI/CD

При каждом пуше в Git нужно автоматически:

1. запустить проверки кода;
2. запустить тесты;
3. проверить сборку backend;
4. проверить сборку frontend;
5. собрать Docker image;
6. при успешном результате обновить приложение в LXC-контейнере Proxmox;
7. перезапустить контейнеры;
8. убедиться, что приложение поднялось.

### 14.2. Рекомендуемый вариант для MVP

Для MVP рекомендуется начать с GitHub Actions или GitLab CI, если репозиторий находится в GitHub/GitLab.

Причина: это проще в настройке, чем отдельный Concourse, не требует отдельного сервера под CI и быстрее даст рабочий результат.

Concourse можно использовать, если хочется отдельную self-hosted CI/CD-систему с декларативными pipeline и хорошей визуализацией. Но для домашнего/семейного проекта он может быть избыточен.

Рекомендация:

- MVP: GitHub Actions/GitLab CI;
- позже, если хочется отдельно потрогать и изучить Concourse — вынести pipeline туда.

### 14.3. Этапы pipeline

Pipeline должен состоять из стадий:

#### 1. Lint и format check

Backend:

- `go fmt` / `gofmt` check;
- `go vet`;
- опционально `golangci-lint`.

Frontend:

- eslint;
- prettier check;
- typecheck, если используется TypeScript.

#### 2. Unit tests

Backend:

```text
go test ./...
```

Frontend:

```text
npm test
```

или другой тестовый раннер.

#### 3. Integration tests

Интеграционные тесты должны поднимать PostgreSQL и проверять:

- создание задачи;
- создание встречи;
- перенос задачи;
- отметку задачи выполненной;
- генерацию login-link;
- работу базовых API endpoint'ов.

Варианты реализации:

- PostgreSQL service внутри CI;
- Testcontainers;
- Docker Compose для тестов.

#### 4. Build check

Backend:

```text
go build ./cmd/app
```

Frontend:

```text
npm run build
```

#### 5. Docker build

Собрать Docker image backend/frontend.

Пример тегов:

```text
registry.example.com/family-planner/app:latest
registry.example.com/family-planner/app:<git-sha>
```

Лучше хранить image в registry:

- GitHub Container Registry;
- GitLab Container Registry;
- Docker Hub;
- private registry.

#### 6. Deploy в LXC Proxmox

После успешной сборки pipeline должен подключиться по SSH к LXC-контейнеру и выполнить:

```text
docker compose pull
docker compose up -d
```

Если image собирается прямо внутри LXC, можно делать `git pull` + `docker compose up -d --build`, но это менее чистый подход. Лучше собирать image в CI, пушить в registry, а внутри LXC только скачивать готовый image.

#### 7. Healthcheck

После деплоя нужно проверить:

```text
GET /health
```

Если приложение не отвечает, pipeline должен показать ошибку.

### 14.4. Окружения

На MVP достаточно одного окружения:

- `production`.

Позже можно добавить:

- `staging` — автоматический деплой из ветки `develop`;
- `production` — деплой из `main` после merge или manual approve.

Рекомендуемая схема веток:

- push в любую ветку: lint + tests + build check;
- pull request / merge request: lint + tests + build check;
- push в `main`: lint + tests + build + docker push + deploy на ВМ.

### 14.5. Secrets

Секреты нельзя хранить в Git.

В CI/CD secrets нужно хранить в настройках CI:

- `TELEGRAM_BOT_TOKEN`;
- `DATABASE_URL` для production, если нужен в deploy;
- `SSH_HOST`;
- `SSH_USER`;
- `SSH_PRIVATE_KEY`;
- `REGISTRY_USERNAME`;
- `REGISTRY_PASSWORD`;
- `JWT_SECRET` или `AUTH_SECRET`;
- `APP_BASE_URL`.

В LXC секреты должны храниться в `.env` файле, который не лежит в репозитории.

### 14.6. Docker Compose в LXC

Пример сервисов:

```yaml
services:
  app:
    image: registry.example.com/family-planner/app:latest
    restart: always
    env_file:
      - .env
    depends_on:
      - postgres
    ports:
      - "8080:8080" # этот порт проксируется через Nginx Proxy Manager

  postgres:
    image: postgres:16
    restart: always
    environment:
      POSTGRES_DB: family_planner
      POSTGRES_USER: family_planner
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

### 14.7. Healthcheck endpoint

Backend должен иметь endpoint:

```text
GET /health
```

Ответ:

```json
{
  "status": "ok",
  "db": "ok",
  "version": "<git-sha>"
}
```

Этот endpoint используется:

- для проверки после деплоя;
- для мониторинга;
- для быстрой диагностики.

## 16. Backup и восстановление

Так как PostgreSQL будет хранить личные семейные данные, нужен backup.

Минимально:

- ежедневный `pg_dump`;
- хранение backup-файлов не только на этой же ВМ, но и во внешнем месте;
- ротация backup'ов, например 7 ежедневных и 4 недельных;
- инструкция по восстановлению.

Пример команды backup:

```text
pg_dump "$DATABASE_URL" > backups/family_planner_$(date +%Y-%m-%d).sql
```

В будущем можно добавить автоматическую проверку восстановления на тестовой базе.
